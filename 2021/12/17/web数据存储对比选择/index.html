web数据存储对比选择

发表于 2021-12-17 分类于 前端
本文字数： 970 阅读时长 ≈ 1 分钟

对目前主流的web存储数据方式进行比较和选择。

结论

使用indexdb，加上其扩展的dexie。

背景

由于近期项目需要进行web存储数据，所以查找资料大致有以下几种实现方式。

Cookies

Cookie是一段不超过4KB的小型文本数据，是某些网站为了辨别用户身份，进行Session跟踪而储存在用户本地终端上的数据（通常经过加密），由用户客户端计算机暂时或永久保存的信息，由一个名称（Name）、一个值（Value）和其它几个用于控制Cookie有效期、安全性、使用范围的可选属性组成。
存储空间太小不适用。

Session

Session对象存储特定用户会话所需的属性及配置信息。这样，当用户在应用程序的Web页之间跳转时，存储在Session对象中的变量将不会丢失，而是在整个用户会话中一直存在下去。session数据会随着会话结束而销毁，并不是一种持久化的本地存储，同时存储大了会消耗内存和磁盘空间。

LocalStorage

LocalStorage可以将存储存储到本地，并且是永久性的存储。并且是使用key-value键值方式存储，但是数据并不是对象，而是字符串的形式，对常见的JSON对象需要进行转换,而且不提供搜索功能，不能建立自定义的索引。同时由于各个浏览器的不同可存储的空间大小并不一致，大概都在5M以内，所以只适用一些小型的数据。

Web SQL

WebSQL也是一种内置在浏览器里存储的技术，属于关系型数据库，使用SQL查询数据，底层采用的sqlite技术支持持久化。由于不是H5的规范一部分，而且并不是所有的浏览器都支持，最重要的是在2010年W3C已经不再支持这种技术。所以拥抱最新的技术吧！

IndexedDB

IndexedDB是HTML5规范里新出现的浏览器里内置的数据库。IndexedDB提供了类似数据库风格的数据存储和使用方式。存储在IndexedDB里的数据是永久保存。IndexedDB 允许储存大量数据，提供查找接口，还能建立索引，从数据库类型来说属于NoSql数据库。特点是键值对存储、异步、支持事务、同源限制、存储空间大和支持二进制存储。

总结

根据项目需求要存储大量的数据，并且避免WebSql被淘汰及其非兼容性，所以最终采用IdexedDB，再加上根据对其扩展封装的API框架dexie.js。
